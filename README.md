# Техническое задание на разработку программного продукта для планирования встреч



## 1. Общие описание
Необходимо разработать программный продукт для планирования встреч сотрудников внутри компании. Система состоит из клиентского мобильного приложения и серверной части.

## 2. Клиентская часть (Android-приложение)
Клиентом является **Android-приложение**, которое должно быть реализовано на **Kotlin** или **Java** с использованием **адаптивной верстки**.

Приложение должно предоставлять интуитивно понятный интерфейс, в котором реализован следующий функционал:

*   **Регистрация и авторизация** сотрудника.
*   **Настройка личного профиля**.
*   **Создание встречи** с возможностью приглашения других сотрудников. Приглашенные сотрудники должны иметь возможность дать бинарный ответ (**принял** / **не принял**).
*   **Просмотр списка активных приглашений** на встречи с возможностью дать ответ о возможности посещения.
*   **Просмотр собственного расписания встреч** в разрезе дня, недели и месяца.

**Важное условие:** временные слоты для встреч разбиты строго по часам. Например, можно создать встречу на 9:00–10:00, но создание встречи на 9:10, 9:05 и т.д. — невозможно.

## 3. Серверная часть
Сервер представляет собой **микросервис на Spring Boot**, работающий с СУБД **PostgreSQL** или **H2**.

Серверное приложение должно:
*   Предоставлять API, необходимый для полноценной работы мобильного приложения.
*   Осуществлять все основные операции с данными в базе через **Spring Data JPA**.
*   Использовать **Liquibase** для создания схемы базы данных и ее предзаполнения.

## User stories
1. Как пользователь, я хочу иметь возможность регистрироваться, чтобы меня могли найти другие пользователи.

2. Как пользователь, я хочу иметь функцию настройки личного профиля, чтобы другие пользователи видели только актуальную информацию обо мне.

3. Как пользователь, я хочу иметь возможность создавать встречи, чтобы другие пользователи могли в них участвовать.

4. Как пользователь, я хочу видеть приглашения и давать на них согласия, чтобы другие пользователи знали о моём решении. 

5. Как пользователь, я хочу видеть запланированные встречи, чтобы не опаздывать на них.

## Use Cases
Цель: регистрация сотрудника
Акторы: сотрудник
Предусловия: у сотрудника есть почта, и он не зарегистрирован в системе
Основной поток:
1. Ввод данных и почты
2. Отправка запроса о регистрации данных на сервер
3. Проверка данных
4. Сохранение данных в базу данных
5. Отправка ответа об успешной регистрации
6. Переход на главный экран
Альтернативный поток:
1. Пользователь вводит некорректные данные и почту
2. Отправка запроса о регистрации данных на сервер
3. Проверка данных
4. Сервер возвращает ошибку регистрации
5. Приложение отображает сообщение о неверных данных пользователю

Цель: редактирование информации в личном профиле
Акторы: сотрудник
Предусловия: у сотрудника есть учётная запись в системе
Основной поток:
1. Сотрудник выбирает пункт о редактировании данных в личном профиле
2. Сотрудник вводит новые данные
3. Отправка запроса об изменении данных на сервер
4. Проверка данных
5. Сохранение данных в базу данных
6. Отправка ответа об успешном изменении данных
7. Вывод сообщения об успешном изменении информации в профиле
Альтернативный поток:
1. Сотрудник вводит некорректные данные
2. Отправка запроса об изменении данных на сервер 
3. Проверка данных 
4. Отправка ответа в приложение о неуспешном изменении данных
5. Приложение выводит сообщение о неуспешном изменении данных

Цель: создание встречи
Акторы: сотрудники
Предусловия: у сотрудника есть учётная запись в системе
Основной поток:
1. Пользователь выбирает пункт о создании встречи
2. Пользователь выбирает участников и вводит данные о встречи
3. Отправка запроса о создании встречи на сервер
4. Сохранение данных о встречи в базу данных
5. Отправка приглашений участникам встречи
6. Отправка ответа об успешном создании встречи
7. Вывод сообщения пользователю об успешном создании встречи
Альтернативный поток:
1. Сервер не смог создать встречу
2. Отправка ответа в приложение о неуспешном создании встречи
3. Вывод сообщения пользователю об неуспешном создании встречи

Цель: узнать согласен или не согласен пользователь на запланированную встречу
Акторы: сотрудники
Предусловие: у сотрудника есть учётная запись, сотруднику пришло приглашение на встречу
Основной:
1. Пользователь открывает список приглашений
2. Пользователь выбирает, что он согласен на приглашение
3. Отправка согласия пользователя на сервер
4. Добавление встречи в список встреч
5. Отправка создателю встречи сообщения о принятии приглашения
Альтернативный поток:
1. Пользователь открывает список приглашений
2. Пользователь выбирает, что он не согласен на приглашение
3. Отправка несогласия пользователя на сервер
4. Удаление пользователя из встречи
6. Отправка создателю встречи сообщения о непринятии приглашения

Цель: получить информацию о запланированных встречах
Акторы: сотрудник
Предусловие: у сотрудника есть учётная запись
Основной поток:
1. Пользователь открывает список встреч
2. Отправка запроса на получение запланированных встреч на сервер
3. Получение списка запланированных встреч из базы данных
4. Отправка списка запланированных встреч в приложение
5. Вывод списка на экран пользователя
Альтернативный поток:
1. Сервер возвращает ошибку
2. Вывод сообщения об ошибки в приложении

Задачи для backend-разработчика:
Реализовать API для регистрации пользователя
Реализовать API для создания встреч
Реализовать API для получения списка встреч
